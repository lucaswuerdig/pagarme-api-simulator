## Shared Decisions
- TS compiles as **CommonJS** (`module: CommonJS`, `moduleResolution: node`) → `dist/`. Relative imports need NO `.js` extension; `require.main === module` is the direct-run guard. Task 08 Vercel shim imports the CJS `app`.
- Express bootstrap lives in `src/server.ts` and exports `app`, `createApp(mountRoutes?)`, `resolvePort(env)`, `start(port?, application?)`, `DEFAULT_PORT` (8088). **`start` gained an optional `application` param (Task 10, defaults to the `app` singleton).** The **direct-run guard** (`require.main===module`, `node dist/server.js`) now builds the app from the Task 07 factory: `start(undefined, createPagarmeApp(createStore()))`, so a directly-run server (Docker, `npm start`) honors `STORE_BACKEND` — NOT just the in-memory default. `server.ts` therefore now statically imports `createStore` from `./store` (no import-time side effects; tests stay hermetic; the guard is `/* c8 ignore */`d).
- **`@vercel/kv` is Upstash HTTP-REST, not raw Redis TCP** — a plain `redis` container cannot back `STORE_BACKEND=kv`. Local KV parity needs an Upstash-REST shim (Task 10's `redis-rest`/`serverless-redis-http`). The KV path is only exercisable locally through that proxy; `memory` is the dependency-free default.
- **Vercel shim (Task 08)**: `api/index.ts` exports `buildApp(env=process.env): Express` = `createPagarmeApp(createStore(env))` and `export default buildApp()`. `@vercel/node` invokes the default-exported Express app directly (it's a `(req,res)` handler) — no `serverless-http`. `vercel.json` rewrites `/(.*)`→`/api`; Vercel passes the ORIGINAL URL so Express still routes `/core/v5/...` (incl. dynamic `charge_id`) unchanged. Env vars (`STORE_BACKEND=kv`, `KV_REST_API_URL`, `KV_REST_API_TOKEN`) documented in `.env.example`; Node version via `engines.node>=20` (no `functions`/runtime block in `vercel.json`). **`tsconfig.json` build still `include:["src"]`** → `tsc`/`dist` never compile `api/` (Vercel's `@vercel/node` does); `tsconfig.test.json` include + `vitest.config.ts` coverage include both extended with `api/` so the shim is typechecked & coverage-counted (100%).
- **App factory (Task 06)**: `server.ts` now exports `createPagarmeApp(store: OrderStore = new InMemoryOrderStore()): Express` — the fully-wired app (calls `createApp((app) => registerRoutes(app, store))` so routes register BEFORE the 404 fallback). `export const app = createPagarmeApp()` ships WITH routes now (no longer bare). **Task 07/08 inject a different backend by passing a store to `createPagarmeApp` — no route changes.** `createApp(mountRoutes?)` is unchanged (still the bare scaffold used by `app.test.ts`).

## Shared Learnings
- Tests live in `tests/unit/` and `tests/integration/`; `npm test` runs `vitest run --coverage` with 80% thresholds enforced in `vitest.config.ts` (coverage scoped to `src/**`). Use `supertest` against the in-process `app`. Build excludes `tests/` (tsconfig `exclude`).
- ESLint flat config (`eslint.config.mjs`, typescript-eslint recommended) does not enable `no-console`. `ban-ts-comment` IS on → `// @ts-expect-error` MUST carry a description.
- **vitest runs via esbuild and does NOT type-check.** Compile-time type assertions (`// @ts-expect-error`, typed fixtures) are inert under `vitest run`. Task 02 added a `typecheck` script (`tsc --noEmit -p tsconfig.test.json`) + `tsconfig.test.json` (rootDir `.`, includes src+tests) so such assertions are actually enforced. Use `npm run typecheck` for any future compile-time type tests; CI (Task 09) should run it alongside lint/build/test.

## Open Risks
- `npm install` reports 6 advisories (4 moderate, 2 critical) all in the **dev-only** vitest/vite/esbuild chain — production runtime depends only on `express`. Do NOT `npm audit fix --force` (it bumps vitest to a 4.x beta = breaking). Relevant to Task 09 CI if it runs `npm audit`.
- **Store-failure → 5xx route mapping is STILL NOT handled (and was NOT added in Task 08).** Task 07's `KvOrderStore` correctly *surfaces* KV failures (errors propagate), but the route layer still has no try/catch to translate a rejected store call into a 5xx — a rejected KV call would currently hang an Express 4 async handler. Task 08 was infra-only (shim + `vercel.json` per its requirements) and did NOT touch routes; this remains an **open follow-up for a route-layer task** (per TechSpec "Error handling conventions"). Caveat for whoever picks it up: an Express 4-arg error middleware trips typescript-eslint `no-unused-vars` on the trailing `_next` — wrap async handlers (`.catch(next)`) or configure `argsIgnorePattern`. (Confirmed in Task 07: the rule also rejects a `_`-prefixed function arg like `_opts`.)
