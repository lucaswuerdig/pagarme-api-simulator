# ADR-005: In-memory store with opaque IDs and a test-only reset route

## Status

Accepted — storage backend superseded by [ADR-006](adr-006.md) for Vercel deployment (Vercel KV
replaces the in-memory `Map`). Opaque IDs and the `/__reset` route remain in force; an in-memory
implementation is retained for local dev and tests.

## Date

2026-06-03

## Context

ADR-001 established that the fake is stateful: it must remember each order/charge so a later
capture or cancel resolves the same `charge_id` and `card.id` and echoes the original amount. Three
realization choices remained open, all flagged as open questions in the PRD:

1. **Where state lives** — pure in-memory, in-memory plus a reset affordance, or durable file
   persistence.
2. **How identifiers are generated** — deterministic sequential IDs (`or_fake_0001`) that tests can
   hardcode, versus opaque unique IDs the test reads back from the create response.
3. **How concurrent test runs on the single shared instance stay isolated.**

These decisions shape the store interface, the response builders, and the documented test pattern.

## Decision

State lives in a **single in-memory `Map` keyed by `charge_id`**, lost on process restart (no
durable persistence in the MVP). Identifiers are **opaque unique IDs** (`or_fake_<rand>`,
`ch_fake_<rand>`, `card_fake_<rand>`, `tran_fake_<rand>`); tests capture the id from the
`POST /core/v5/orders` response and reuse it for subsequent capture/cancel calls rather than
hardcoding it. A **test-only `POST /__reset` route** clears the store so a suite can start from a
known-empty state on the shared instance. `/__reset` is explicitly NOT a Pagar.me route and is
documented as a homologation helper.

## Alternatives Considered

### Alternative 1: Pure in-memory, no reset route

- **Description**: In-memory Map only; state clears solely on restart.
- **Pros**: Strictest YAGNI; zero non-Pagar.me surface.
- **Cons**: No clean way to isolate suites on a shared always-on instance without restarting it.
- **Why rejected**: A one-line reset route materially improves shared-instance test isolation at
  negligible cost and risk.

### Alternative 2: Sequential deterministic IDs

- **Description**: Monotonic counters per prefix (`or_fake_0001`, `0002`, …) so tests can hardcode
  expected IDs.
- **Pros**: Assertable IDs without reading the create response.
- **Cons**: Only deterministic on a freshly reset instance; counters advance with concurrent
  traffic, making hardcoded assertions flaky on the shared box.
- **Why rejected**: Opaque IDs are robust under concurrency; the consuming app already reads the id
  from the response (`getIdentifier`), so hardcoded sequential IDs are unnecessary.

### Alternative 3: File persistence

- **Description**: Flush the Map to a JSON file so state survives restart.
- **Pros**: Survives restarts.
- **Cons**: Adds file I/O, write-debounce, and concurrency concerns for a benefit the PRD marks
  optional (Phase 2).
- **Why rejected**: YAGNI for the MVP; in-memory is sufficient and state may be safely discarded on
  restart per ADR-001.

## Consequences

### Positive

- Minimal store with a clear interface (`create`, `get`, `update`, `clear`).
- Opaque IDs avoid cross-test coupling and collisions under concurrent runs.
- The reset route gives suites a deterministic clean slate without restarting the instance.

### Negative

- State is lost on restart (acceptable per ADR-001).
- `/__reset` is a small surface that does not exist on the real Pagar.me; it must be clearly
  documented as test-only and never relied on by the consuming app.

### Risks

- A misused `/__reset` during a live shared run could wipe another suite's in-flight state.
  Mitigation: document it as a suite-setup helper and keep it out of the Pagar.me-compatible path;
  it can be gated behind an env flag if abuse appears.

## Implementation Notes

Store interface lives in `src/store/orderStore.ts`. ID generation is a small helper
(`src/util/ids.ts`) producing prefix + random suffix. `/__reset` clears the Map. The same
`charge_id` and `card.id` minted at order creation are returned consistently across the order's
lifecycle so capture/cancel resolve.
