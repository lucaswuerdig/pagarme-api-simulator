# ADR-006: Deploy on Vercel serverless functions with Vercel KV for lifecycle state

## Status

Accepted

## Date

2026-06-03

## Context

The fake will be deployed on **Vercel**, and the project must be ready for a GitHub pipeline to ship
it there. Vercel runs **stateless serverless functions** with cold starts and potentially many
concurrent instances. The current design ([ADR-005](adr-005.md)) keeps order/charge lifecycle state
in an **in-memory `Map`** on a single always-on instance. On Vercel that `Map` is not reliably shared
across invocations: a `capture` or `cancel` request can land on a different function instance than the
original `POST /core/v5/orders` and find no record — which breaks the core promise of
[ADR-001](adr-001.md) (coherent sale → capture → cancel). A state backend that survives across
stateless invocations is required, and the Express app must be exposed as a Vercel function.

## Decision

Deploy the fake as a **single Vercel serverless function that wraps the existing Express app**:
`api/index.ts` exports the app and `vercel.json` rewrites `/(.*)` to `/api`, so all `/core/v5/...`
routes (including the dynamic `charge_id` path) reach the same handler unchanged.

Lifecycle state moves from the in-memory `Map` to **Vercel KV (Upstash Redis)**. The `OrderStore`
interface from [ADR-005](adr-005.md) is unchanged; a `KvOrderStore` stores each `OrderRecord` under a
prefixed key (`ch:<chargeId>`) with a TTL (default 24h). An `InMemoryOrderStore` is retained for local
dev and tests, selected by a `STORE_BACKEND` env var (`kv` on Vercel, `memory` locally/CI). Opaque IDs
and the test-only `POST /__reset` route from ADR-005 remain; `/__reset` deletes keys under the `ch:`
prefix (scan + delete), never `flushall`, to stay safe on a shared store. Docker/compose is retained
for **local development only** ([ADR-004](adr-004.md)); Vercel is the deployment target.

## Alternatives Considered

### Alternative 1: Self-encoding charge IDs (stateless, no store)

- **Description**: Encode outcome + amount + card into the `charge_id` itself so capture/cancel decode
  it and rebuild a coherent response with no external store.
- **Pros**: Zero infrastructure; truly serverless-native; no per-request store latency.
- **Cons**: Long client-held IDs that carry state; partial capture/refund accounting (`canceled_amount`
  across multiple calls) is awkward without a record; further diverges the ID aesthetic.
- **Why rejected**: KV preserves the clean stateful model and the existing `OrderStore` interface with
  less conceptual churn and easier partial-amount accounting.

### Alternative 2: Keep in-memory (best-effort)

- **Description**: Leave the `Map`; accept that coherence holds only within a warm instance.
- **Pros**: No infrastructure.
- **Cons**: Lifecycle coherence is not guaranteed across requests on Vercel.
- **Why rejected**: Breaks ADR-001's core promise on the chosen platform.

### Alternative 3: Per-route Vercel function files

- **Description**: Split each endpoint into its own `api/` file using Vercel file-based routing.
- **Pros**: More idiomatic Vercel.
- **Cons**: Fragments the shared store/resolver/builders and complicates the dynamic
  `/charges/[id]/capture` path; more boilerplate.
- **Why rejected**: A single function wrapping Express keeps the design intact with minimal change.

## Consequences

### Positive

- Lifecycle stays coherent across stateless invocations and cold starts via shared KV.
- Minimal change: Express, routes, resolver, builders, and the `OrderStore` interface are unchanged.
- CI stays hermetic — tests run against `InMemoryOrderStore`, no live KV needed.

### Negative

- Adds a managed dependency (Vercel KV / Upstash) and env vars (`KV_REST_API_URL`, `KV_REST_API_TOKEN`).
- Introduces network latency per store operation and a KV provisioning step in the Vercel project.

### Risks

- Concurrent test runs share one KV namespace and could interleave. Mitigation: opaque per-order IDs,
  prefixed keys, TTL expiry, and `/__reset` (prefix-scoped, not `flushall`).
- Leaked KV credentials. Mitigation: store as Vercel/GitHub encrypted env vars; rotate as needed.

## Implementation Notes

Dependencies: `@vercel/kv`, `serverless-http` (or Express's native handler export). Files: `api/index.ts`,
`vercel.json` (rewrites). Store selection via `STORE_BACKEND` (`kv` | `memory`). This decision
supersedes the storage backend of [ADR-005](adr-005.md) and refines the deployment/packaging of
[ADR-004](adr-004.md) (Docker becomes local-dev-only). ADR-001's coherence goal is preserved; only its
realization changes from one in-memory instance to serverless functions over shared KV.
