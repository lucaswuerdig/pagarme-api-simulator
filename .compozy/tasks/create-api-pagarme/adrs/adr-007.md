# ADR-007: GitHub Actions CI/CD deploying to Vercel via the Vercel CLI

## Status

Accepted

## Date

2026-06-03

## Context

The project must be ready for a **GitHub pipeline that deploys to Vercel** ([ADR-006](adr-006.md)).
Two patterns achieve this: let Vercel's Git integration auto-deploy (with GitHub Actions only running
tests), or have a GitHub Actions workflow own the deploy via the Vercel CLI. We must pick one so the
repository ships with a working, reproducible pipeline.

## Decision

Add a GitHub Actions workflow (`.github/workflows/ci.yml`) with two jobs:

1. **test** — runs on pushes and pull requests: `npm ci`, lint, and `vitest` (unit + integration
   against `InMemoryOrderStore`).
2. **deploy** — `needs: test`, runs only on push to `main`: deploys with the Vercel CLI
   (`vercel pull --environment=production`, `vercel build --prod`, `vercel deploy --prebuilt --prod`),
   authenticated via the `VERCEL_TOKEN`, `VERCEL_ORG_ID`, and `VERCEL_PROJECT_ID` GitHub secrets.

Pull requests may optionally produce a Vercel **preview** deployment (`vercel deploy` without `--prod`).
Vercel's own Git auto-deploy is left disabled so the workflow file is the single source of truth for
how and when the service ships.

## Alternatives Considered

### Alternative 1: Vercel Git integration + CI gate

- **Description**: Connect the repo to Vercel; Vercel auto-deploys (prod on `main`, preview on PRs).
  GitHub Actions only runs lint + tests as a status check.
- **Pros**: Less in-repo deploy configuration; preview URLs out of the box.
- **Cons**: Deploy logic lives in the Vercel dashboard, not the repository; less explicit and harder to
  audit/reproduce from the codebase.
- **Why rejected**: The requirement is an explicit GitHub pipeline that deploys to Vercel; owning the
  deploy in the workflow keeps it visible and reproducible.

## Consequences

### Positive

- Deployment is reproducible and reviewable in-repo; tests gate every production deploy.
- Optional PR preview deployments aid review.

### Negative

- Vercel tokens/IDs must be managed as GitHub secrets.
- Vercel CLI version drift must be watched.

### Risks

- A leaked `VERCEL_TOKEN` enables unauthorized deploys. Mitigation: GitHub encrypted secrets, a
  least-privilege token, and branch protection on `main`.

## Implementation Notes

Workflow at `.github/workflows/ci.yml`. Secrets: `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`.
Node version pinned via `actions/setup-node`. The `deploy` job builds with `vercel build` so the
prebuilt output is uploaded with `--prebuilt`, keeping build logic identical between CI and Vercel.
