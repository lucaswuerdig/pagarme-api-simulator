# ADR-003: Credit-card-only scope with deterministic magic-card outcomes

## Status

Accepted

## Date

2026-06-03

## Context

Pagar.me v5 supports multiple payment methods (credit card, PIX, boleto). The fake exists to remove
the test dependency on the homologation environment, and the consuming app's most fragile,
highest-volume test surface is the credit-card flow. We must set two scope boundaries: which payment
methods the fake covers, and how a test author selects the outcome (approved, declined, transaction
error, order failed, gateway unavailable).

## Decision

The fake covers **credit-card flows only** — the five routes in the specification — with **no
support planned for PIX or boleto**. Outcomes are selected **deterministically by the card number
(or `card_id`/`card_token`)** via a fixed magic-card table. There is no runtime override mechanism
and no admin-defined scenarios.

## Alternatives Considered

### Alternative 1: Include PIX and boleto

- **Description**: Extend the fake to also reproduce PIX and boleto order/charge flows.
- **Pros**: Broader coverage of the consuming app's payment surface.
- **Cons**: Larger contract surface (QR codes, boleto lines, asynchronous settlement, webhooks) for
  flows that are not the source of the current test pain.
- **Why rejected**: Out of scope by explicit decision; the credit-card flow is the target problem.

### Alternative 2: Runtime outcome override (header or control endpoint)

- **Description**: Keep magic cards but also allow forcing any outcome via a special header or a
  control endpoint, for tests that cannot vary the card number.
- **Pros**: Flexibility when the card number is fixed by the test fixture.
- **Cons**: Adds a non-Pagar.me control surface and a second source of truth for outcomes.
- **Why rejected**: The consuming app already controls the card number sent, so magic cards alone
  cover the required scenarios without diverging from the real API's request shape.

## Consequences

### Positive

- Minimal, well-bounded contract; deterministic and reproducible outcomes.
- The fake's request/response shapes stay identical to the real Pagar.me API (no custom controls).

### Negative

- Tests that need a specific outcome must use the corresponding magic card number/token.
- Any future PIX/boleto testing remains dependent on the real homologation environment.

### Risks

- A needed scenario might not map to a magic card. Mitigation: the magic-card table is the single
  documented source of scenarios and can be extended with new reserved numbers if a gap appears.

## Implementation Notes

The magic-card table from the specification is the canonical scenario list. The same `charge_id`
and `card.id` must be returned consistently across an order's lifecycle so capture/cancel resolve.
