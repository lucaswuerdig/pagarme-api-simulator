# ADR-002: Deliverable boundary — fake service only; consuming-app integration as a guide

## Status

Accepted

## Date

2026-06-03

## Context

The specification's Section 7 lists changes the consuming Laravel application needs in order to
point at the fake during homologation: a configurable base URL in `config/pagarme.php`, a
`setApiUrl()` that reads it, and a `PAGARME_API_URL` entry in the test `.env`. That application
lives in a **separate repository**; it is not present in this project.

We must decide whether this effort's deliverable includes those cross-repo code changes or only the
fake service itself.

## Decision

This effort delivers **only the standalone fake Pagar.me API service** (this repository). The
Section-7 integration is delivered as a precise, copy-paste **connection guide** documented
alongside the fake. Applying those changes is owned by the consuming-application team in their own
repository.

## Alternatives Considered

### Alternative 1: Apply the integration changes cross-repo

- **Description**: Treat the `config/pagarme.php`, `setApiUrl()`, and `.env` edits as in-scope
  deliverables, coordinating commits into the consuming app's repository.
- **Pros**: End-to-end wiring done in one effort.
- **Cons**: Couples this project to another repository's release process, review, and ownership;
  broadens scope and risk beyond building the fake.
- **Why rejected**: Cross-repo ownership and release coordination add risk disproportionate to the
  small, well-specified edits, which the owning team can apply from a guide.

### Alternative 2: Service only, no integration guidance

- **Description**: Ship the fake and leave integration entirely to the other team to figure out.
- **Pros**: Narrowest possible scope.
- **Cons**: Leaves a discoverability/adoption gap; the connection steps are subtle (URL-swap not
  key-swap, `config:clear`, never set in production).
- **Why rejected**: A short connection guide materially de-risks adoption at negligible cost.

## Consequences

### Positive

- Clear ownership boundary; this repository stays self-contained and independently shippable.
- Adoption is still smooth because the connection guide removes guesswork.

### Negative

- Final end-to-end wiring depends on the consuming-app team applying the documented changes.

### Risks

- The consuming app could be misconfigured to point at the fake in production. Mitigation: the
  guide mandates leaving `PAGARME_API_URL` unset in production (default falls back to the real
  Pagar.me URL) and is explicit that this is a test/homologation-only switch.

## Implementation Notes

The connection guide reproduces Section 7 of the specification verbatim as documentation; no code
is committed to the consuming repository from this effort.
