# ADR-001: Stateful, contract-faithful fake over stateless stub

## Status

Accepted

## Date

2026-06-03

## Context

The consuming application's credit-card test suites depend on the Pagar.me v5 homologation
environment, whose availability is unreliable. We are building a fake Pagar.me API to remove that
dependency. The behavioral contract (routes, request shapes, response fields, and the body-based
approval rules) is fully defined in the project specification.

The open product decision is the fidelity level of the fake: should it merely return canned
per-request responses, or should it track the lifecycle of each order/charge so that multi-step
flows (sale → capture → cancel/refund) remain coherent? The fake will run as a shared, always-on
homologation instance the whole team points to.

## Decision

Build a **stateful, contract-faithful fake** (Approach A). It implements exactly the five
credit-card routes, decides outcomes deterministically from the incoming card number/token (magic
cards), and maintains per-order/charge lifecycle state so a later capture or cancel resolves the
same `charge_id`/`card.id` and echoes the original amount. It exposes a basic health check so the
homologation environment can confirm the instance is live.

## Alternatives Considered

### Alternative 1: Minimal stateless stub (Approach B)

- **Description**: The same five routes, but responses are canned purely from the incoming card
  number with no memory of previously created orders or charges.
- **Pros**: Fastest to build; nothing to persist; trivially horizontally scalable.
- **Cons**: Cannot faithfully reproduce sale → capture → cancel coherence; capture/cancel cannot
  verify the charge ever existed or return its real amount; weaker for refund and multi-step tests.
- **Why rejected**: The consuming app exercises capture and cancel/refund against charges created
  earlier; without lifecycle state those flows can only be faked superficially, reducing test value.

### Alternative 2: Fake plus inspection/admin layer (Approach C)

- **Description**: Approach A plus a request-inspection surface, runtime state reset, and dashboards
  for the shared instance.
- **Pros**: Strong QA visibility into traffic hitting the shared box.
- **Cons**: Exceeds the stated need; because scenario selection is fixed to magic cards, the admin
  layer is mostly observability rather than control.
- **Why rejected**: YAGNI — observability tooling is not required to remove the homologation
  dependency and can be added later if a concrete need emerges.

## Consequences

### Positive

- Full credit-card lifecycle behaves coherently against a single shared instance.
- Deterministic outcomes make test assertions stable and reproducible.
- Faithful to the specification, so the consuming app needs no behavioral compromises.

### Negative

- Requires lightweight state-keeping (in-memory is sufficient) and basic operational care for the
  shared instance.

### Risks

- A shared instance accumulates state across many concurrent test runs. Mitigation: key state by
  the deterministic identifiers in the request and avoid cross-test coupling; persistence is
  optional and state may be safely discarded on restart.

## Implementation Notes

In-memory state indexed by `charge_id` is sufficient; durable persistence is optional. The
technical realization (language, framework, storage) is deliberately left to the TechSpec.
