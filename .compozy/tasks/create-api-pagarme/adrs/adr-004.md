# ADR-004: Node.js + Express + TypeScript implementation stack

## Status

Accepted

## Date

2026-06-03

## Context

The project specification (`_idea.md`) deliberately leaves the implementation stack open
("A stack é livre — Node/Express, PHP/Laravel, Python/FastAPI, etc."), requiring only that the
service honor the Pagar.me v5 route, request, and response contracts. The repository is greenfield:
no source files, no build tooling, no language committed.

The fake is a small HTTP service: five routes, JSON in/JSON out, in-memory state, and a
body-shape contract where many fields the consuming app reads later (the ⭐ fields) must be present
and correctly typed. It will run as a shared, always-on homologation instance the team points to in
place of the Pagar.me sandbox. We must select a concrete stack so the TechSpec can specify
structure, build, and packaging.

## Decision

Implement the fake in **Node.js with Express, written in TypeScript**. TypeScript models the
Pagar.me v5 `Order`, `Charge`, `Transaction`, `Card`, and `Token` shapes as types, giving
compile-time protection over the contract fields the consuming app depends on. Express provides the
five `/core/v5/...` routes plus the health and test-only reset endpoints. The build is `tsc` to a
`dist/` directory; the service runs as `node dist/server.js`.

## Alternatives Considered

### Alternative 1: Go (net/http + chi)

- **Description**: Single static binary, chi router, struct-based response models.
- **Pros**: Tiny distroless image, fast startup, minimal runtime dependencies; ideal for an
  always-on shared instance.
- **Cons**: More ceremony for ad-hoc JSON shapes; team chose JS/TS familiarity over binary size.
- **Why rejected**: Deployment-size advantage did not outweigh the team's preference for the
  JavaScript ecosystem for a simple JSON fake.

### Alternative 2: PHP/Laravel

- **Description**: Match the consuming application's own stack.
- **Pros**: Same language the payments team maintains.
- **Cons**: Heavy runtime (php-fpm + nginx) for a trivial fake; slower cold start; more moving
  parts to host.
- **Why rejected**: Disproportionate runtime weight for a five-route stub; the fake is independent
  of the consuming app and need not share its stack.

### Alternative 3: Python/FastAPI

- **Description**: FastAPI with Pydantic models and auto-generated OpenAPI docs.
- **Pros**: Concise, strong request validation.
- **Cons**: No advantage over TypeScript here, and the fake intentionally does little request
  validation (it trusts the consuming app's pre-validated payloads per `_idea.md` §4.1).
- **Why rejected**: TypeScript delivers the same contract-typing benefit within the chosen
  JavaScript ecosystem.

## Consequences

### Positive

- Typed contract models reduce the risk of omitting or mistyping the ⭐ fields the consuming app
  reads downstream.
- Express keeps the five-route surface explicit and easy to read against the spec.
- Large ecosystem for testing (vitest, supertest) and packaging.

### Negative

- Adds a `tsc` build step versus plain JavaScript or a single Go binary.
- Requires a Node runtime in the deployed image.

### Risks

- Type models could drift from the real Pagar.me contract. Mitigation: types mirror `_idea.md`
  exactly, and integration tests assert the ⭐ fields are present in responses.

## Implementation Notes

Project layout: `src/server.ts` (Express app + bootstrap), `src/routes/` (orders, charges, tokens,
health, reset), `src/store/` (in-memory order/charge store), `src/magic/` (card→outcome table),
`src/types/pagarme.ts` (contract models), `src/responses/` (response builders). Build with `tsc`,
run with `node dist/server.js`, port from `PORT` (default `8088`).
