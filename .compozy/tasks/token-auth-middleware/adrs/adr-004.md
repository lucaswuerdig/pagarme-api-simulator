# ADR-004: Test authentication via a shared helper and a committed homologation test token

## Status

Accepted

## Date

2026-06-09

## Context

With the gate always on (ADR-001), every request to a protected route must carry a
valid token. Codebase exploration found **44 supertest calls** to protected routes
across four test files (`tests/unit/routes.test.ts`, `tests/integration/httpRoutes.test.ts`,
`tests/integration/vercelHandler.test.ts`, `tests/integration/docsAccuracy.test.ts`),
and **no shared supertest wrapper** — each test calls `request(app)` directly. The
old `RESET_SECRET` tests in `tests/unit/routes.test.ts` (which exercised the
`x-reset-secret` guard) must also be replaced, since ADR-002 removes that mechanism.

We must decide how tests obtain and send a valid token without scattering the secret
or letting new tests silently forget it.

## Decision

Add a single committed **homologation test token** to the `VALID_TOKENS` allowlist —
an obviously-fake value such as `test_token` — and a small shared test helper
(`tests/helpers/authedRequest.ts`) that wraps a supertest agent and sets
`Authorization: Basic base64("test_token:")` on every request. All 44 protected-route
call sites are refactored to use the helper. The helper imports the token from
`src/auth/tokens.ts` so there is a single source of truth and no drift.

The removed `RESET_SECRET` tests are replaced by tests asserting the new behavior:
`/__reset` returns 401 without a token and 204 with a valid token.

## Alternatives Considered

### Alternative 1: Reuse an existing allowlist token (no dedicated test entry)

- **Description**: Tests import whatever token already exists in the list.
- **Pros**: No extra entry.
- **Cons**: Couples tests to a "production" homologation value; removing/rotating that
  value breaks tests for an unrelated reason.
- **Why rejected**: A dedicated, clearly-fake `test_token` decouples tests from the
  real homologation set.

### Alternative 2: Per-call `.set(...)` at each of the 44 sites, no helper

- **Description**: Add the header inline everywhere.
- **Pros**: No new helper module.
- **Cons**: 44 edits; a new test can silently omit the header and the omission is easy
  to miss in review.
- **Why rejected**: A helper centralizes auth and makes "authenticated request" the
  default, lowering the chance of future drift.

### Alternative 3: Injectable allowlist per test

- **Description**: Make the middleware accept an injected allowlist so tests pass their
  own token set.
- **Pros**: Full isolation from the committed list.
- **Cons**: Extra wiring; diverges from the cards precedent, where tests just use known
  card numbers against the module constant.
- **Why rejected**: YAGNI; importing the committed `test_token` is simpler and matches
  how card-scenario tests already work.

## Consequences

### Positive

- One helper makes authenticated requests the default; negative tests still send a
  bogus token (or none) explicitly.
- Single source of truth for the test token; no magic strings duplicated across files.

### Negative

- A clearly-fake token is committed in the allowlist (acceptable — the whole fake is a
  homologation double, like the committed magic card numbers).

### Risks

- **A test file bypasses the helper.** Mitigation: refactor all current sites and note
  the helper in the testing section so it is the obvious default for new tests.

## Implementation Notes

- Helper shape: `authedRequest(app)` returns a thing whose `.post/.get/.delete`
  pre-set the `Authorization` header; or a thin `withAuth(req)` applied per call.
- Negative-path tests (missing/invalid token → 401) deliberately bypass the helper.
- The `vercelHandler.test.ts` and `docsAccuracy.test.ts` builders use the same
  `registerRoutes` path, so the helper works uniformly across all app builders.
