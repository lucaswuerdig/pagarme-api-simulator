# ADR-003: Token-auth middleware design â€” src/auth module, mount order, and 401 shape

## Status

Accepted

## Date

## Context

ADR-001 chose a static in-file token allowlist with an always-on Basic-auth gate;
ADR-002 extended that gate to `POST /__reset` and retired `RESET_SECRET`. This ADR
records the implementation-level design: where the code lives, how the middleware
parses the credential, where it mounts in the Express pipeline, and the 401 body
shape.

Relevant facts from codebase exploration:
- No Basic-auth or base64 parsing exists today â€” this is greenfield.
- The magic-card precedent (`src/magic/cards.ts`) keeps pure, deterministic data and
  logic in a module separate from the Express routes that consume it.
- `registerRoutes` (`src/routes/index.ts`) mounts, in order: `healthRouter`,
  `resetRouter`, then the `/core/v5` `logRequests` middleware and routers.
- Express runs middleware in registration order; a router that has already matched a
  path cannot be guarded by middleware registered after it.
- The repo's existing 401 (old reset guard) and 404 fallback both use
  `{ error, message }`.

## Decision

**Module layout.** Add a new `src/auth/` directory with two files:
- `src/auth/tokens.ts` â€” pure, Express-free: the `VALID_TOKENS` allowlist (a
  `ReadonlySet<string>` / readonly array mirroring `MAGIC_CARD_NUMBERS`), an
  `isValidToken(token)` predicate, and a `parseBasicToken(header)` helper that
  decodes `Authorization: Basic base64("<token>:")` and returns the token (or
  `undefined` when the header is absent/malformed).
- `src/auth/middleware.ts` â€” a thin Express middleware `requireToken` that calls the
  pure helpers and either calls `next()` or responds `401`.

**Parsing.** `parseBasicToken` reads `req.headers.authorization`, requires the
`Basic ` scheme, base64-decodes the rest, and takes the substring before the first
`:` as the token. Anything missing or malformed yields `undefined` â†’ 401.

**Mount order.** In `registerRoutes`, register `requireToken` **after**
`healthRouter()` and **before** `resetRouter()`:
```
app.use(healthRouter());     // open
app.use(requireToken);       // guards everything below
app.use(resetRouter(store));
app.use("/core/v5", logRequests);
app.use("/core/v5", ordersRouter(store)); // + charges, tokens
```
This guards `/__reset` and all `/core/v5` routes (and any unknown path â†’ 401 before
the 404 fallback), while `/health` stays open because it is matched first.

**401 body.** Respond `{ error: "unauthorized", message: "..." }`, consistent with the
old reset guard and the 404 fallback.

## Alternatives Considered

### Alternative 1: Single file or no new directory

- **Description**: Put allowlist + middleware in one `src/auth.ts`, or place the
  allowlist in `src/magic/` and the middleware in `src/routes/`.
- **Pros**: Fewer files / no new directory.
- **Cons**: Mixes pure logic with Express wiring (hurts unit testing), or mixes auth
  tokens into the card domain.
- **Why rejected**: The two-file `src/auth/` split mirrors the cards/routes
  separation and keeps `isValidToken`/`parseBasicToken` trivially unit-testable.

### Alternative 2: Mount the gate after the reset router (per-prefix on /core/v5 only)

- **Description**: Guard only `/core/v5` and keep a separate guard on `/__reset`.
- **Pros**: Narrower middleware scope.
- **Cons**: Requires two guards; contradicts ADR-002's single-model goal; if mounted
  after `resetRouter`, `/__reset` would not be guarded at all.
- **Why rejected**: A single `requireToken` mounted before both routers is simpler and
  correct.

## Consequences

### Positive

- One small, well-isolated module; pure logic is unit-testable without HTTP.
- A single mount point governs the entire protected surface.
- 401 shape is consistent with the rest of the API.

### Negative

- Unknown paths now return 401 (no token) instead of 404; acceptable and arguably
  better (does not reveal which routes exist). Any 404-on-unknown-path test must send
  a valid token to still observe a 404.

### Risks

- **Malformed-header edge cases** (no scheme, non-base64, missing colon). Mitigation:
  `parseBasicToken` returns `undefined` for all of them and is covered by unit tests.

## Implementation Notes

- Mirror `cards.ts`: `export const VALID_TOKENS` as the canonical list; keep it the
  single source of truth.
- `requireToken` has signature `(req, res, next)`; it reads the module-level
  `VALID_TOKENS` (not injected), matching how routes read the card maps.
- Never log the full token; if any diagnostic is added, mask it.
