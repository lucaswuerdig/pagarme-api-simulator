# ADR-002: Unify access control under the token gate, retiring RESET_SECRET

## Status

Accepted

## Date

2026-06-09

## Context

Before this feature the only access guard in the codebase was the optional
`RESET_SECRET` on `POST /__reset` (Issue 005): when the `RESET_SECRET` env var is
set, the reset helper requires a matching `x-reset-secret` header, otherwise it stays
open. It exists because the fake is a shared always-on instance and an
unauthenticated global clear lets any caller wipe another suite's in-flight orders.

ADR-001 introduces an always-on token allowlist gate over the `/core/v5` business
routes. That leaves a choice for the destructive `/__reset` helper: keep its separate
`RESET_SECRET` mechanism, or fold it under the new token gate. Running two parallel
auth models (a header secret for reset, a Basic token for everything else) is
redundant and harder to reason about.

## Decision

Extend the token-validation gate to also cover `POST /__reset`, and **retire the
`RESET_SECRET` / `x-reset-secret` mechanism**. After this change a caller proves
authorization the same way everywhere — by presenting a valid token from the
allowlist via `Authorization: Basic`. `GET /health` remains open as a liveness probe.

## Alternatives Considered

### Alternative 1: Keep RESET_SECRET alongside the new token gate

- **Description**: Leave `/__reset` guarded by its own `RESET_SECRET` header and apply
  the token gate only to `/core/v5`.
- **Pros**: No change to the existing reset guard or its tests.
- **Cons**: Two distinct auth models and two secrets to manage; the destructive
  endpoint protected differently from the rest of the surface.
- **Why rejected**: The stakeholder chose to cover `/__reset` with the token gate; a
  single auth model is simpler and more consistent.

### Alternative 2: Leave /__reset open and gate only /core/v5

- **Description**: Drop `RESET_SECRET` and do not protect `/__reset` at all.
- **Pros**: Least code.
- **Cons**: Reinstates the Issue 005 risk — any anonymous caller can wipe the shared
  store.
- **Why rejected**: Unacceptable for the shared instance; the destructive endpoint
  needs at least the same protection as the business routes.

## Consequences

### Positive

- One auth model across the whole protected surface; one kind of credential to
  manage and document.
- The destructive reset endpoint is protected by the same always-on gate, closing the
  Issue 005 risk without a separate secret.

### Negative

- The `RESET_SECRET` env var and `x-reset-secret` header are removed; any caller or
  script that relied on them must switch to sending a valid token.

### Risks

- **Existing reset callers break.** Mitigation: the change is documented in the README
  and connection guide, and the reset tests are updated to authenticate with a token.

## Implementation Notes

- Removing `RESET_SECRET` supersedes the relevant part of the Issue 005 design; the
  reset router no longer reads that env var.
- The token middleware must run before both the `/core/v5` routers and the reset
  router so a single mount point governs all protected routes.
