# ADR-001: Static in-file token allowlist with an always-on Basic-auth gate

## Status

Accepted

## Date

2026-06-09

## Context

The fake Pagar.me API is deployed as a single, always-on shared instance that the
whole team points at during homologation. Today every request carries the real
Pagar.me credential as `Authorization: Basic base64("<token>:")`, but the fake
**accepts and ignores it** — the `/core/v5` routes are completely open to anyone
who knows the URL.

Two needs motivate adding validation: (1) **access control** — block callers that
do not present a known token from using the shared instance; and (2) **fidelity** —
let the consuming app exercise its own `401` handling path, mirroring how the real
Pagar.me rejects an invalid key.

The project already has a precedent for storing fixed test data in a committed
source file: the "magic cards" allowlist in `src/magic/cards.ts`, where card
numbers and token ids map to deterministic outcomes. The stakeholder asked for the
valid tokens to live in a file "the same way it was done for the cards."

The `_idea.md` contract (§3.3) constrains the response model: business outcomes
(success, refusal, business error) MUST return HTTP 200, because a 4xx makes the
consuming app's Guzzle client throw and fall into infra-error handling instead of
business parsing. The same section explicitly permits HTTP 401 for an
authentication error — so the auth gate is the one place a 4xx is correct.

## Decision

Add a token-validation middleware in front of the protected routes, backed by a
**static, committed allowlist file** holding an array of valid tokens (mirroring
`src/magic/cards.ts`). The middleware reads the `Authorization: Basic` header,
decodes it, extracts the token (the username portion, before the empty password),
and checks membership in the allowlist. A request with a missing, malformed, or
non-listed token is rejected with **HTTP 401**; a request bearing a listed token
proceeds unchanged.

Validation is **always on** in every environment. The existing test suite and local
development are updated to send a valid token from the allowlist. Adding or rotating
a token is a deliberate code change plus redeploy, exactly like editing the magic
cards.

## Alternatives Considered

### Alternative 1: File allowlist + environment-variable override

- **Description**: Same committed file, but an env var (e.g. on Vercel) can extend
  or override the list so tokens are added without a code change.
- **Pros**: Rotate or add tokens without a redeploy.
- **Cons**: Two sources of truth for "what is a valid token"; more wiring and
  surface for misconfiguration. The stakeholder explicitly asked for a fixed
  in-file list.
- **Why rejected**: YAGNI for a fixed homologation token set; the in-file list is
  the single, auditable source of truth and matches the cards precedent.

### Alternative 2: Presence-only (passthrough) validation

- **Description**: The gate only requires a well-formed `Basic` token to be present,
  without comparing it against any list.
- **Pros**: Simplest possible; existing tests barely change.
- **Cons**: Any non-empty string passes, so it provides no real access control and
  does not satisfy the "array of valid tokens" requirement.
- **Why rejected**: Fails the primary access-control goal; the stakeholder wants a
  concrete allowlist.

### Alternative 3: Optional gate, enabled only when configured (RESET_SECRET style)

- **Description**: Validate only when an env flag is set or the list is non-empty,
  leaving the instance open otherwise — mirroring the optional `RESET_SECRET` guard.
- **Pros**: Backwards compatible; the existing test suite is untouched.
- **Cons**: The deployed shared instance could be left open by accident; weaker
  guarantee for the access-control goal. With a fixed committed list the "open"
  state never really applies.
- **Why rejected**: The stakeholder chose always-on enforcement for a stronger and
  more predictable guarantee.

## Consequences

### Positive

- Unauthorized callers can no longer use the shared homologation instance.
- The consuming app can exercise its real `401` path by sending an unlisted token —
  no special affordance needed.
- The valid-token set is a single, auditable, in-repo source of truth, consistent
  with the magic-cards pattern the team already understands.

### Negative

- Every existing automated test and local-dev invocation must now send a valid
  token, so the test suite and any local scripts need a one-time update.
- Adding or rotating a token requires a code change and redeploy.

### Risks

- **Token leakage in logs.** Mitigation: never log the full token; the existing
  request logger already strips secrets, and any token reference in new logs is
  masked (as the consuming app already does with `substr`/`str_repeat`).
- **Lockout from a bad list edit.** Mitigation: the allowlist is covered by tests,
  and CI runs the suite (which must authenticate) before any production deploy.

## Implementation Notes

- The middleware mounts ahead of the protected routers (the `/core/v5` block and the
  reset helper); see ADR-002 for the reset scope. `/health` stays open as a liveness
  probe.
- The 401 body shape (repo `{error, message}` convention vs. an exact Pagar.me
  envelope) is an implementation detail to be settled in the TechSpec; the consuming
  app's Guzzle treats any 4xx as a thrown exception regardless of body.
- Mirror the cards file structure: a dedicated module exporting the token array and a
  pure `isValidToken` helper, kept separate from the Express wiring so it is unit
  testable with a plain list.
